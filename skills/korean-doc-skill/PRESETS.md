# 문서 유형별 구성

아래 구조는 필수 양식이 아니라 필요한 섹션을 고르는 목록이다. 원문에 실제 내용이 있을 때만 섹션을 사용한다(SKILL.md 규칙 3). 원문 용어를 존중하되 번역투와 낯선 표현은 그대로 보존하지 않는다.

## 자연어 요청과 문서 유형 연결

- `전략 문서`, `방향성 검토`, `논의 안` -> Strategy Memo
- `사업 기회 검토`, `시장 검증`, `파트너 연계 검토`, `사업성 검토`, `MVP 사업 검토` -> Business Opportunity / Market Validation
- `상위기획`, `연간 계획`, `컨셉 기획` -> Upper Planning / Concept Planning
- `상세기획`, `PRD`, `기획서` -> PRD / Service Planning Doc
- `유즈케이스`, `직군별 시나리오` -> Use Case / Vertical Workflow Doc
- `허브 문서`, `인덱스 페이지`, `문서 모음 소개` -> Hub / Index Page
- `위클리`, `주간 업데이트` -> Weekly Update. 상태판이 큰 보드형이면 Team Weekly Board
- `이슈 대응 문서`, `Q&A 대응`, `답변 정리` -> Response Memo
- `이슈 히스토리`, `장애 이력`, `대응 이력` -> Issue History / Incident Log
- `킥오프`, `추진 보고`, `추진 방향`, `프로젝트 보고`, `임원 공유` -> Executive Kickoff / 추진 보고
- `실행 가이드`, `점검 절차`, `런북`, `운영 가이드`, `스크립트 사용법`, `체크리스트` -> Execution Guide / Runbook

## 기본 구성 (맞는 문서 유형이 없을 때)

1. `목적`
2. `핵심 요약`
3. 본문 분석·판단·요청 섹션
4. `리스크 / 전제`
5. `다음 액션 / R&R`

상단 요약은 판단·방향·다음 액션을 3개 이내의 불릿으로 정리한다. 본문 분석도 판단형 제목 아래 `핵심 수치 / 해석 / 확인 필요`처럼 독립적인 핵심을 2~3개 불릿으로 드러낸다. 원인과 해석이 이어질 때만 2문장 이내의 짧은 문단을 쓴다. `개요` 성격의 섹션에는 가장 큰 변화·목적·범위만 두고 예외와 구현 상세는 본문으로 내린다.

## Strategy Memo

구성: `목적` / `핵심 요약` / `왜 중요한가` / `판단 / 방향` / `근거` / `리스크 / 전제` / `시사점`

- 제목에 판단을 담고 첫 화면에 왜 중요한가 + 판단 + 시사점 표시
- 선택지 중심 문서는 첫 화면에 `권고안`, `비교 기준`, `요청사항`을 두고, 간결한 비교표 하나와 상세 재무·운영 근거를 아래에 배치
- 분석 결과는 판단형 제목 아래 핵심 수치·의미·확인할 점을 2~3개 불릿으로 정리하고, 산식·집계 기준·예외는 펼치기나 부록으로 이동

## Upper Planning / Concept Planning

구성: `목적` / `핵심 요약` / `왜 지금` / `방향` / `우선순위` / `1차 범위` / `후속 상세기획 연결` / `리스크 / 전제`

- 실행계획보다 방향을 정하는 문서. 첫 화면에 방향 / 우선순위 / 1차 범위 / 결정 필요사항 표시
- 실행 상세와 긴 이력은 아래로 내리고 방향 / 우선순위 / 후속 상세기획에서 풀 내용을 구분

## Business Opportunity / Market Validation

구성: `추진 목적 및 사업 비전` / `목표 시장 및 매출 가능 규모 검토` / `시장 수요 및 자사 경쟁력` / `MVP 방향성 및 사용자 시나리오` / `사업 모델 및 과금 구조 검토` / `법무/개인정보보호` / `확인 필요 사항` / `참고 문서`

- 임원 보고서가 아니라 사업화 가능성을 확인하는 검토 문서. `핵심 요약`이나 `핵심 판단`을 억지로 만들지 않음
- 첫 화면에 확인할 사업 기회, 협력 대상, 첫 시장, 확장 방식 표시
- 사실과 가정을 구분하고 `가정`, `MAX`, `참고용`, `N/A`, `확인 필요`로 표시. 거친 시장 규모는 확정 매출이 아님
- 근거 없는 수익성 계산을 앞세우지 않음. 비용 항목, 최소 ARPU, 이익률 위험까지만 확인하고 단말 기능·기존 권한·무료 채널로 거의 들지 않을 비용은 `확인 필요`로 이동
- 법무·개인정보보호 흐름은 검토 전 기획 가정으로 취급. 자동 수집, 동의, 보관, 삭제, 접근 기록, 광고성 메시지 위험을 필수 준수 조건으로 명시
- 반복 필드는 표로 정리: 시장 단계 / 목표 고객 / 규모 / 매출 가능 규모, AS-IS / TO-BE, 우선순위 / 기능 / 근거. 우선순위 표의 상태는 `핵심` / `추가` / `필수 준수`로 표시 가능

## PRD / Service Planning Doc

구성: `목적` / `배경` / `문제 정의` / `목표` / `범위` / `주요 사용자 흐름` / `정책/예외` / `리스크` / `오픈 이슈`

- 적용 범위와 제외 대상을 명확히 구분하고, 미결 항목은 별도 섹션에 둔다. 첫 화면에 방향 / 적용 범위 / 제외 대상 / 미결 항목을 표시
- 정책 비중이 큰 문서는 기본 규칙 / 예외 / 우선 적용 규칙 / 변경 이력을 불릿이나 표로 분리하고 한 문장에 합치지 않음. 정확한 필드명인 `override`만 원문 유지. 순서는 활성화 조건 -> 생성/저장 -> 예외 -> 업로드/리마인드 -> 미노출/파기 -> 오픈 이슈가 유용
- 화면·배치·노출 방식 변경은 상단에서 AS-IS / TO-BE 비교
- 큰 매핑표는 상세 내용을 유지하고 앞에 짧은 안내 추가
- 변경 이력은 상단 요약 아래에 배치

## Skill Spec / Capability Spec

구성: `한 줄 정의` / `언제 쓰는가` / `입력` / `출력` / `핵심 동작` / `예외/금지` / `평가 기준`

- 긴 설명보다 정확한 정의와 짧은 예시를 우선하고 홍보 문구는 제외
- 작성 규칙·가이드는 긴 문단보다 짧은 라벨 블록(`A. 기본 방향`, `B. 문체`) 우선

## Report / Decision Material

구성: `목적` / `핵심 요약` / `주요 판단` / `옵션 비교` / `리스크` / `일정/R&R`

- 요약을 먼저 두고 핵심 주장은 본문에 유지. 접기(expand)는 상세 근거·계산 자료에 사용하고 본문 자체는 접지 않음
- 첫 화면은 목적 또는 요청 결정 1문장과 핵심 판단·권고안·다음 액션 3개 이내 불릿으로 구성. 본문도 긴 설명문보다 판단형 제목과 병렬 불릿을 우선

## Executive Kickoff / 추진 보고

구성: `핵심 요약` / `추진 개요` (목적·범위) / `추진 계획·일정` / `진행 현황·주요 판단` / `다음 단계·핵심 원칙`

- 프로젝트 시작·중간을 임원·비실무자에게 공유하는 문서: 결과 보고가 아니라 추진 틀·방향 합의가 목적
- 최상단 `핵심 요약`은 3~4불릿 박스 (무엇을 하는 프로젝트·현재 단계·문서 목적·요청 사항). 처음 보는 임원이 첫 화면에서 전체 그림을 잡게
- 상위·차주 보고를 동반하면 승인·요청사항을 명시. 보고 일정은 프로젝트 단계와 별개로 분리 표기
- 추진 계획·일정은 단계 표 + 타임라인 시각화, 단계별 상태(완료/진행 중/예정) 라벨
- 주요 판단은 주제명이 아니라 판단형 제목으로 작성 (`### 원가는 A·B에 편중, 단기 개선은 두 영역에 집중`)
- 본문은 핵심 주장으로 간결히, 상세 근거·배경은 펼치기(expand)로 접어 깊이 확보 (REFERENCE 참조)
- 데이터는 확정(실측)과 가설(추정·POC)을 구분 표기

## Execution Guide / Runbook / Script Usage Guide

구성: `실행 요약` / `바로 실행` / `판정 기준` / `결과 확인` / `권한 제한 시 수동 확인` / `주의사항`

- 설명문이 아니라 실행 지원 문서. 최종 산출물·파일·열·판정 지점을 첫 화면에 배치
- 가장 짧은 성공 경로부터 제시. 필요한 것이 파일·명령·결과 필드 하나뿐이면 상단에 명시
- 권한·환경 문제 시 사용할 대체 방법은 본 흐름과 섞지 않고 별도 섹션으로 분리
- CloudShell·콘솔·스크립트·운영 점검 문서는 `A. 실행 요약 -> B. 바로 실행 -> C. 판정 기준 -> D. 결과 확인 -> E. 수동 확인` 순서

## Weekly Update

구성: `이번 주 핵심 변화` / `진행 현황` / `이슈 / 리스크` / `다음 주 계획` / `지원 필요사항`

- 기본 위클리는 섹션당 한 화면을 넘기지 않고 완료 / 막힘 / 다음 작업을 분리
- 첫 화면에 핵심 변화 / 해야 할 일 / 리스크를 상세 표보다 먼저 배치
- 간결한 요약 아래에 고정 프로젝트 표를 유지하고 빈 양식 행과 시스템 메시지는 먼저 삭제

## Team Weekly Board

구성: `이번 주 핵심 신호` / `주요 공유 / 결정 필요` / `핵심 To-Do` / `포트폴리오 보드` / `리스크 / 지원 요청` / `참고 링크 / 부록`

- 상태판 비중이 큰 위클리는 표 위에 간결한 요약 하나를 두고 기존 표 구조 유지
- 전주 진행 / 금주 계획 / 장기 목표가 섞인 셀을 먼저 분리하고 첫 화면을 회의에서 바로 읽기 쉽게 정리

## Decision One-Pager

구성: `목적` / `추천안` / `선택지 비교` / `리스크 / 전제` / `요청 결정사항`

- 추천안을 상단에 두고 선택지가 같은 기준으로 비교될 때만 비교표 사용. 결정에 영향을 주는 배경 이력만 유지

## Use Case / Vertical Workflow Doc

구성: `한 줄 정의` / `왜 이 직군 / 상황` / `대표 장면` / `AI가 해야 하는 동작` / `사용자 가치` / `기본 기능` / `사용자 설정값` / `제한 / 금지`

- `상황 -> 동작 -> 가치` 흐름을 장면 중심 제목으로 구성
- 유즈케이스가 여러 개면 같은 형식의 병렬 블록 사용. 반복 필드가 있는 장면이 3개 이상이면 고정 블록이나 표로 정리

## Hub / Index Page

구성: `한 줄 정의` / `왜 이 문서 묶음` / `현재 핵심 문서` / `문서 구조 / 카테고리` / `기본 원칙`

- 상단에 문서 묶음의 역할, 핵심 문서, 분류 방식 표시
- 묶음 불릿을 기본으로 사용하고 같은 문서 목록을 여러 형식으로 반복하지 않음

## Tracking / Ops Board

구성: `현재 상태` / `이번 주 핵심 변화` / `주요 이슈 / 리스크` / `트래킹 항목` / `다음 액션` / `지원 필요사항`

- 진행 중인 현황판은 상태·담당자·다음 작업·시점을 반복 필드 표로 관리. 상태 라벨은 `진행 중`, `대기`, `완료`, `리스크`로 통일
- 현재 상태 / 결정 필요 / 참고 메모를 분리하고 미결 사항과 완료 이력을 섞지 않음
- 종료된 이슈의 시간순 기록이 중심이면 Issue History / Incident Log 사용

## Issue History / Incident Log

구성: `문서 범위` / `최근 핵심 이슈` / `이슈 인덱스` / `사건별 기록` / `공통 패턴 / 재발 방지` / `참고 링크`

- 현재 상태판이 아니라 과거 이력을 찾는 문서. 사건별 시작·종료 시점과 시간순 흐름을 명확히 표시
- 사건별로 `이슈 내용 / 원인 / 영향 / 조치 / 후속` 형식을 고정하고 상단에 간결한 색인 배치
- 진행 중인 이슈 현황판이 중심이면 Tracking / Ops Board 사용

## Response Memo

구성: `목적` / `핵심 입장` / `주요 질문 / 이슈` / `권고 답변` / `리스크 / 확인 필요사항` / `요청사항 / 다음 액션`

- PR 답변·임원 Q&A·외부 문의 준비 문서는 공식 입장이나 권고 답변을 첫 화면에 배치
- 공식 답변 / 내부 판단 / 미확정 사항 / 추가 확인 필요를 분리. 질문 / 답변 / 리스크 / 담당이 반복될 때만 질문표 사용
- 날짜별 대응 이력이 중심이면 Issue History / Incident Log 사용
