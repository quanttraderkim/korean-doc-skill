# review checklist

사용 방법은 단순합니다. 먼저 [`rubric.md`](./rubric.md)로 점수를 잡고, 그다음 아래 체크리스트로 어떤 부분이 왜 좋거나 나쁜지 메모합니다. 반복 실패는 [`failure-taxonomy.md`](./failure-taxonomy.md) 코드와 함께 기록하는 것을 권장합니다.

## 1. 첫 화면

- 독자와 문서의 실제 용도(판단·분석·정의·실행·추적·참고)가 분명한가
- 요청자를 자동으로 독자로 두지 않고, 실제 공유·발표 대상이 이해하고 판단할 수 있게 썼는가
- `요청하신`, `검토했습니다`, `자료를 확인한 결과`처럼 에이전트가 작성자에게 작업 과정을 보고하는 문장이 남지 않았는가
- 판단 문서는 결론·요청, 분석 문서는 주요 결과, 실행 문서는 시작 조건·절차, 참고 문서는 범위·탐색 기준이 먼저 보이는가
- 문서를 왜 읽어야 하는지 바로 이해되는가
- 전략·제안 문서라면 현재 사실에서 사업상 의미와 제안으로 이어지는 설명이 보이는가

## 2. 문체

- `~입니다 / ~합니다 / ~됩니다`가 과도하지 않은가
- 지나치게 기계적인 `~함` 반복도 없는가
- `~한다 / ~이다 / ~됐다` 문장이 3개 이상 이어지는 문단이 없는가
- 명사형 / 짧은 판단형 문체가 자연스러운가
- 독립적인 판단·수치·근거·조건·액션이 짧은 병렬 불릿으로 드러나는가
- 원인과 해석처럼 연결이 필요한 문단은 2문장 이내인가

## 3. 용어

- 영어를 먼저 정한 뒤 한국어 단어로 일대일 번역한 표현이 없는가
- 같은 영어라도 역할 구분·적용 범위·판정 기준처럼 문맥에 맞게 풀었는가
- 정확한 필드명·공식 명칭이 아닌 일반 문장에 `정본`, `경계값`, `보안 경계`, `오너`, `드라이버`, `가시성` 같은 직역투가 남지 않았는가
- DAU·API처럼 널리 쓰이는 말과 낯선 자작 용어를 구분했는가
- 제품명·필드명·기술 식별자를 정확히 유지했는가
- 검색되지 않거나 설명이 필요한 새 약어를 만들지 않았는가
- 영어 원문을 몰라도 뜻이 바로 이해되는가
- 동료가 실제 회의에서 그대로 쓸 표현인가

## 4. 구조

- 제목이 섹션 역할에 맞는가. 전략·분석 결과는 확인된 결과나 의미를 보여주고, PRD·런북·참고 위키는 정책·행동·조회 대상을 바로 찾게 하는가
- 데이터 분석에서 방법·집계 조건·한계가 결과 제목을 대신하지 않는가. 방법·품질 자체가 문서 목적이면 주제형 제목을 유지했는가
- 각 섹션이 서로 다른 독자의 질문에 답하는가
- 같은 판단·근거·확인사항이 다른 섹션·표·펼치기에 반복되지 않는가
- 하나의 결정만 다루는 리더·임원용 장을 구현·권한·QA·지표·운영 소제목으로 다시 쪼개지 않았는가
- 기존 문서의 한 장 안에 `목적`·`핵심 요약`을 다시 만들어 본문 판단을 반복하지 않았는가
- `먼저 볼 항목`, `지금 먼저 볼 것`, `읽기 순서` 같은 메타 제목을 기본 섹션명으로 쓰지 않았는가
- 긴 문단이 적절히 쪼개져 있는가
- 기존 번호 체계를 유지했는가. 번호가 탐색에 도움이 될 때만 추가했고 번호를 맞추기 위해 소제목을 만들지 않았는가

## 5. 표와 불릿

- 표가 실제 비교·결정을 문장이나 불릿보다 빠르게 만드는가
- 실제 열이 있다는 이유만으로 실무 체크리스트를 표로 만들거나, 같은 질문에 답하는 표를 여러 개 두지 않았는가
- `항목 / 설명`만 있는 표를 불릿으로 바꾸는 편이 더 자연스럽지 않은가
- 상단 요약의 `항목 / 판단`, `구분 / 내용` 2열 표가 단순 라벨+설명인데 표로 남아 있지 않은가
- Confluence/wiki에서 중첩불릿의 라벨 역할 1단계 불릿만 `####` 제목으로 승격됐는가
- 불릿이 병렬 정보에만 쓰였는가
- 하위 설명 불릿이나 병렬 항목 자체를 과하게 제목으로 승격하지 않았는가
- 줄바꿈만 많은 가짜 개괄식이 되지 않았는가
- 기본 규칙, 예외 처리, 우선 적용 규칙, 변경 이력이 한 문장에 뭉치지 않았는가

## 6. 분량 / 절제

- 리더·임원용 본문에 판단과 관계없는 구현·필드·QA·운영 세부가 남지 않았는가. 반대로 PRD·런북·참고 문서에서 필요한 세부를 삭제하지 않았는가
- 별도 실무 문서로 옮긴 세부 항목을 본문에서 다시 목록으로 나열하지 않았는가
- 불릿 수를 맞추기 위해 관련 내용을 잘못 합치거나 섹션을 인위적으로 나누지 않았는가
- 불릿 한 개에 서로 무관한 포인트가 섞이지 않았는가
- 제목이나 다른 불릿을 반복하는 불릿이 없는가
- 내용 없는 섹션을 프리셋 채우기식으로 만들지 않았는가
- 늘어난 내용이 정확성·실행 안전성·탐색성 개선에 실제로 필요한가. 새 해석이나 운영 규칙을 근거 없이 추가하지 않았는가
- 파일 경로·커밋·검증 명령·자료 수령 과정·수정 횟수처럼 작업 기록에 둘 내용이 공유 본문에 남지 않았는가
- 분석 결과를 바꾸는 한계와 독자에게 필요 없는 작성 과정의 불확실성을 구분했는가
- 근거가 없어 삭제한 초안의 주장이나 제안에만 쓰이던 수치·반박 설명이 남아, 독자에게 없던 문제를 다시 부각하지 않는가. 공식 결정 기록이라면 실제 선택지의 제외 근거가 보존됐는가
- 발표·보고 자료의 후속 데이터 목록은 독자가 직접 확보·승인해야 할 때만 남겼는가. 데이터가 없어 결론이 제한되면 자료 목록 대신 판단할 수 없는 범위만 설명했는가
- 발표·보고용 문서라면 후속 분석 실행에만 필요한 데이터와 추출 방법을 작업 기록·인수인계로 옮겼는가. 분석 실행 런북·참고 문서에 필요한 절차는 보존했는가
- 펼치기가 중복·불필요한 세부를 삭제하지 않고 보관하는 곳으로 쓰이지 않았는가

## 7. Confluence 포맷 안정성

- inline code 앞뒤 공백이 Confluence export에서 붙지 않는가
- underscore가 들어간 식별자가 markdown emphasis처럼 깨지지 않는가
- 정확한 필드명이 꼭 필요하지 않은 곳은 `요금표`, `우선 적용 규칙`, `캠페인 ID`처럼 읽히는 라벨을 썼는가
- 리스트 뒤에 표나 다음 제목이 붙을 때, export/rendering 후에도 블록 경계가 분명한가

## 8. 문서 타입 적합성

- 전략 메모답게 판단과 근거가 보이는가
- 데이터 분석이라면 확인된 결과가 방법·집계 조건·한계보다 먼저 보이고 분모와 해석 조건은 가까이 남아 있는가
- 데이터 분석의 비중·집중도·상관관계가 비교 기준이나 품질·성과 근거 없이 과다 사용·비효율·원인·개선 과제로 바뀌지 않았는가
- 사업 기회 / 시장 검증 문서라면 목적, 1차 시장, 확장 모델, MVP, 과금, 법무/개인정보보호, 확인 필요사항이 분리돼 있는가
- PRD답게 범위와 오픈 이슈가 분리돼 있는가
- 유즈케이스 문서라면 `상황 / 동작 / 가치 / 제한` 구조가 보이는가
- 허브 문서답게 문서 묶음의 역할과 핵심 문서가 보이는가
- 상위기획 / 컨셉 기획이라면 방향, 가정, 우선순위가 먼저 보이는가
- skill spec이라면 입력 / 출력 / 예외 / 평가 기준이 분리돼 있는가
- 주간 업데이트라면 진행 / 이슈 / 다음 액션이 짧게 분리돼 있는가
- 런북이라면 순서 / 명령 / 판정값 / 중단 조건이 정확하며 원문에 없는 운영 규칙을 만들지 않았는가
- 참고 위키·데이터 카탈로그라면 출처 / 필드 / 정의 / 조인·적용 기준 / 제한사항을 찾기 쉬운가
- 원페이저라면 요청 결정사항이 분명한가

## 9. 가정 / 비용 / 법무

- 아직 검증되지 않은 매출, 원가, 법무 가능성을 단정하지 않았는가
- 참고용 규모와 실제 매출 가능 규모를 구분했는가
- 단말 기능, 기존 권한, 무료 채널 등으로 비용이 낮을 수 있는 항목을 불필요하게 비용 리스크로 키우지 않았는가

## 10. 최종 판단

- 그대로 써도 되는가
- 구조만 조금 더 손보면 되는가
- 모드 자체를 source-preserving에서 report-first로 바꿔야 하는가
