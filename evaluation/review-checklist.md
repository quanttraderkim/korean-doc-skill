# review checklist

사용 방법은 단순합니다. 먼저 [`rubric.md`](./rubric.md)로 점수를 잡고, 그다음 아래 체크리스트로 어떤 부분이 왜 좋거나 나쁜지 메모합니다. 반복 실패는 [`failure-taxonomy.md`](./failure-taxonomy.md) 코드와 함께 기록하는 것을 권장합니다.

## 1. 첫 화면

- 목적이 바로 보이는가
- 결론, 요청사항, 다음 액션이 바로 보이는가
- 문서를 왜 읽어야 하는지 바로 이해되는가

## 2. 문체

- `~입니다 / ~합니다 / ~됩니다`가 과도하지 않은가
- 지나치게 기계적인 `~함` 반복도 없는가
- 설명문이 길게 `~한다`체로 흘러가지 않는가
- 명사형 / 짧은 판단형 문체가 자연스러운가

## 3. 구조

- 제목이 메시지 역할을 하는가
- 긴 문단이 적절히 쪼개져 있는가
- 긴 문서는 `A. / A-1.` 넘버링이 필요한가
- 기존 문서가 숫자형 검토 자료라면 `1. / 3-1.` 구조를 무리 없이 유지했는가

## 4. 표와 불릿

- 표가 비교 축, 반복 필드, 소유자, 리스크 같은 목적에 맞게 쓰였는가
- `항목 / 설명`만 있는 표를 불릿으로 바꾸는 편이 더 자연스럽지 않은가
- 불릿이 병렬 정보에만 쓰였는가
- 줄바꿈만 많은 가짜 개괄식이 되지 않았는가
- 기본 규칙, 예외 처리, override, 변경 이력이 한 문장에 뭉치지 않았는가

## 5. Confluence 포맷 안정성

- inline code 앞뒤 공백이 Confluence export에서 붙지 않는가
- underscore가 들어간 식별자가 markdown emphasis처럼 깨지지 않는가
- 정확한 필드명이 꼭 필요하지 않은 곳은 `Rate Card`, `priority override`처럼 읽히는 라벨을 썼는가
- 리스트 뒤에 표나 다음 제목이 붙을 때, export/rendering 후에도 블록 경계가 분명한가

## 6. 문서 타입 적합성

- 전략 메모답게 판단과 근거가 보이는가
- 사업 기회 / 시장 검증 문서라면 목적, 1차 시장, 확장 모델, MVP, 과금, 법무/개인정보보호, 확인 필요사항이 분리돼 있는가
- PRD답게 범위와 오픈 이슈가 분리돼 있는가
- 유즈케이스 문서라면 `상황 / 동작 / 가치 / 제한` 구조가 보이는가
- 허브 문서답게 어디부터 읽을지 보이는가
- 상위기획 / 컨셉 기획이라면 방향, 가정, 우선순위가 먼저 보이는가
- skill spec이라면 입력 / 출력 / 예외 / 평가 기준이 분리돼 있는가
- 주간 업데이트라면 진행 / 이슈 / 다음 액션이 짧게 분리돼 있는가
- 원페이저라면 요청 결정사항이 분명한가

## 7. 가정 / 비용 / 법무

- 아직 검증되지 않은 매출, 원가, 법무 가능성을 단정하지 않았는가
- 참고용 규모와 실제 매출 풀을 구분했는가
- 단말 기능, 기존 권한, 무료 채널 등으로 비용이 낮을 수 있는 항목을 불필요하게 비용 리스크로 키우지 않았는가

## 8. 최종 판단

- 그대로 써도 되는가
- 구조만 조금 더 손보면 되는가
- 모드 자체를 source-preserving에서 report-first로 바꿔야 하는가
