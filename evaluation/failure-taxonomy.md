# failure taxonomy

반복 실패를 자유서술로만 남기면 누적이 어렵습니다. 아래 코드로 먼저 분류하고, 필요할 때만 설명을 덧붙이는 방식 권장입니다.

## 구조 / 첫 화면

- `F1` 목적이 첫 화면에 안 보임
- `F2` 결론 / 요청사항 / 다음 액션이 상단에서 안 보임
- `F3` 제목이 해당 섹션에서 무엇을 얻는지 보여주지 못함
- `F4` 긴 문단이 구조 없이 이어짐
- `F5` 넘버링이 필요한 문서인데 구조 표지가 약함
- `F26` 숫자형 검토 문서에 맞는 `1. / 3-1.` 구조를 불필요하게 `A. / A-1.`로 바꿈
- `F36` `먼저 볼 항목`, `읽기 순서` 같은 메타 제목을 반복 사용함
- `F46` 리더·임원용 보고 본문에 판단·승인과 관계없는 구현 체크리스트·필드 목록·상세 QA·운영 절차를 유지함
- `F52` 요청자를 실제 독자로 간주해 `요청하신`, `필요하시면` 같은 대화형 문구로 공유 문서를 작성함
- `F53` 도구·파일·검토 순서·수정 횟수·에이전트 다음 작업처럼 작성 과정에 해당하는 내용을 공유 문서 본문에 넣음
- `F54` 무엇이 미확인이고 어떤 판단이 불가능한지 밝히지 않은 채 `자료가 제한적`, `추가 검토 필요` 같은 방어 문구를 남김

## 문체

- `F6` `~입니다 / ~합니다 / ~됩니다` 공손체 과다
- `F7` 기계적 `~함` 반복으로 부자연스러움
- `F8` `~한다 / ~이다 / ~됐다` 문장이 3개 이상 이어지고, 독립적인 판단·수치·조건·액션이 불릿 없이 한 문단에 묻힘
- `F9` 추상 수식어 과다

## 용어 / 번역투

- `F43` 정확한 필드명·공식 명칭이 아닌 일반 문장에서 영어와 한국어를 일대일 대응해 `경계값`, `보안 경계`, `정본`, `진실 소스` 같은 어색한 표현을 사용함
- `F44` DAU·API·제품명·필드명처럼 널리 쓰이거나 정확히 보존해야 하는 용어를 억지로 번역해 뜻이 달라짐
- `F45` 통용되지 않는 명사·약어를 새로 만들어 별도 설명 없이는 뜻을 알 수 없음

## 표 / 불릿

- `F10` 병렬 정보가 아닌데 불릿 남발
- `F11` 표가 필요한 비교/요약 정보가 문장에 묻힘
- `F12` 줄바꿈만 많고 실제 개괄식이 아님
- `F24` 비교 축이 없는데 표를 과하게 사용함
- `F25` `핵심 판단` 섹션을 필요 이상으로 반복함
- `F31` 기본 규칙 / 예외 / 우선 적용 규칙 / 변경 이력이 한 문장에 뭉침
- `F35` 상단 요약의 단순 `항목 / 판단` 표를 라벨+설명 구조로 바꾸지 않음
- `F37` Confluence/wiki에서 중첩불릿의 라벨 역할 1단계 불릿을 `####` 제목으로 승격하지 않음
- `F38` 하위 설명 불릿이나 병렬 항목 자체까지 제목으로 과하게 승격함
- `F49` 실제 열은 있지만 독자의 비교·결정에 도움이 없거나, 인접 표와 같은 질문을 반복하는 표를 사용함

## 분량 / 절제

- `F39` 라벨 섹션에 우선순위 없이 항목이 쌓여 요약 역할을 못함
- `F40` 내용이 없는데 프리셋 섹션(`시사점`, `리스크`, `보고 시 설명 포인트` 류)을 템플릿 채우기식으로 생성함
- `F41` 정확성·안전성·탐색성 개선 없이 제목·요약·설명을 보태 문서가 불필요하게 길어짐
- `F42` 불릿이 제목이나 다른 불릿을 다른 표현으로 반복함, 또는 한 불릿에 두 생각이 뭉침
- `F47` 같은 판단·근거·확인사항을 여러 섹션·표에서 이름만 바꿔 반복함
- `F48` 삭제하거나 별도 실무 문서로 옮길 중복·불필요 세부를 펼치기에 보관함

## Confluence 포맷

- `F32` inline code 앞뒤 공백이 붙거나 underscore가 markdown emphasis처럼 깨짐
- `F33` 정확한 필드명이 필요하지 않은데 코드형 식별자를 과하게 사용함
- `F34` 리스트 뒤 표나 다음 제목이 붙어 블록 경계가 모호해짐

## 문서 타입 적합성

- `F13` 전략 메모인데 판단보다 설명이 앞섬
- `F14` PRD인데 범위 / 오픈 이슈 분리가 약함
- `F15` 유즈케이스 문서인데 `상황 / 동작 / 가치 / 제한` 구조가 약함
- `F16` 허브 문서인데 읽는 순서와 카테고리 역할이 안 보임
- `F17` 원페이저인데 요청 결정사항이 희미함
- `F27` 사업 기회 / 시장 검증 문서인데 목적, 시장, MVP, 과금, 법무, 확인사항 분리가 약함
- `F21` 이슈 히스토리 / 인시던트 로그인데 사건 / 원인 / 영향 / 조치 / 후속 구조가 약함
- `F22` 대응 문서인데 질문 / 답변 / 리스크 / 확인 필요사항 분리가 약함
- `F23` live tracking board와 historical log가 한 문서 안에서 혼합됨

## 원문 보존 / 재작성 정도

- `F18` 원문을 과하게 축약해 논리 손실
- `F19` report-first가 아닌데 과도하게 재구성
- `F20` source-preserving이어야 하는데 새 해석이 과도하게 들어감

## 제목 역할 불일치

- `F50` 분석 결과 문서에서 방법·집계 조건·한계가 확인된 결과를 밀어내고 주요 제목을 차지함
- `F51` PRD·런북·참고 위키의 제목까지 판단형으로 바꿔 정책·절차·정의를 찾기 어려움

## 가정 / 비용 / 법무

- `F28` 참고용 시장 규모를 실제 매출 추정처럼 과하게 계산함
- `F29` 아직 검토 전인 비용이나 법무 가능성을 단정함
- `F30` 단말 기능, 기존 권한, 무료 채널로 낮출 수 있는 비용을 불필요하게 리스크로 키움

코드는 꼭 여러 개를 붙여도 됩니다. 예를 들어 `F3 + F8 + F15`처럼 기록하면 한 번의 실패가 어느 층위에서 발생했는지 더 분명해집니다.
